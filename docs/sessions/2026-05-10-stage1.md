4. **책 검색 + Edge Function** (Dart 아키텍트 + 기획자 + QA) — 캐시 사전조회 + ON CONFLICT DO UPDATE + 400ms debounce

## 결정 (DECISIONS.md 외 본 세션 한정 흐름 기록)

자세한 결정 근거는 `DECISIONS.md` 2026-05-10 섹션 참고. 본 세션의 자문 → 매니저 판단 흐름:

- **폰트 weight**: 자문가 둘 다 정적 OTF 권고 → 매니저가 NotoSerifKR 가변 폰트 채택 (25 MB 절감 + 모든 weight 무료, 디자이너 Regular 누락 우려도 자동 해소)
- **BottomNav 탭 수**: 아키텍트 5탭 vs 기획자 3탭+[+] → 기획자 채택 (인터뷰 데이터 근거, 친구 탭 cold start 함정)
- **`/book/:id` 인증**: 기획자 게스트 미리보기 채택 (deep-link install funnel)
- **Edge Function 도입 시점**: QA 권고로 Stage 4 → Stage 1 앞당김 (키 노출·rate limit 보호)
- **카카오 V1.5 미룸**: GoTrue가 `account_email` scope를 강제 요청, 카카오 개인 앱은 비즈 인증 없이 거부
- **books upsert 충돌**: QA 권고로 `ON CONFLICT (isbn13) DO UPDATE` (메타 갱신용)
- **마이그레이션 명명**: 14자리 timestamp `YYYYMMDDHHMMSS_*.sql` (Supabase CLI 표준)

## 알게 된 함정 (다음 사람 안 빠지게)

1. **Kakao 개인 앱은 `account_email` scope 등록 불가** — 비즈니스 인증 받기 전엔 동의항목 page에서 회색. Supabase GoTrue가 이 scope를 강제 요청해서 KOE205 발생. 우회: 비즈 인증 또는 `kakao_flutter_sdk` + `signInWithIdToken`
2. **Supabase Redirect URLs `**` 와일드카드는 custom scheme에 안 먹음** — `io.github.tgparkk.bookquote://**`는 매칭 실패, **정확한 URL** 등록 필요
3. **`flutter install`은 매번 uninstall + install** — SharedPreferences (Supabase 세션 저장소) wipe됨. dev엔 `adb install -r build/app/outputs/flutter-apk/app-debug.apk` 사용
4. **Supabase 무료 SMTP 시간당 2건 제한** — dev 검증 5번이면 1시간 대기. 처음부터 Resend 같은 외부 SMTP 권장
5. **Resend free tier sandbox는 가입 이메일에만 발송 가능** — 다른 이메일은 `unexpected_failure` 반환. 베타 친구 5명 발송하려면 도메인 verify 필요
6. **Mobile deep link `io.github.tgparkk.bookquote://auth/callback` Uri 파싱** — Dart는 host=`auth`, path=`/callback`으로 쪼갬. go_router는 path만 보니 라우터에 `/callback` 별도 추가 필요
7. **Supabase 마이그레이션 파일명** — 14자리 timestamp 강제 (`YYYYMMDDHHMMSS_*.sql`). 우리는 처음 `YYYYMMDD_NNN_*.sql` 썼다가 CLI가 같은 timestamp로 인식해서 rename 필요
8. **Riverpod 3.x에서 `StateProvider`, `AutoDispose*` 클래스 제거** — Notifier + AsyncNotifier 패턴으로 전환. 기존 가이드 자료 다수가 구버전 기준
9. **Flutter web 기본 hash URL 전략** — magic link redirect URL과 충돌. `usePathUrlStrategy()` 호출 필수
10. **`palette_generator` discontinued** — V1엔 그대로 사용, V2에서 Flutter 내장 `ColorScheme.fromImageProvider` 마이그레이션 검토

## 매니저 회고

**잘 동작한 패턴**:
- 자문가 충돌 → 매니저 종합 → 사용자 승인 흐름이 매끄러움. 디자이너가 코드를 실제로 읽고 의견 낸 점이 결정적이었음 (폰트 weight 케이스)
- QA 자문가가 P0 함정 (`refreshListenable` trap, books race condition) 짚어준 게 출시 직전 디버깅 회피 효과
- 외부 셋업은 사용자에게 위임하되 **단계별 클릭 위치까지 정확히** 안내한 게 효율적

**개선할 점**:
- Kakao OAuth는 매니저가 사전에 "개인 앱 + Supabase 충돌" 알고 있어야 했음 (자문 단계에서 미발견 → 사용자 시간 소비)
- Resend sandbox 제한도 사전 안내 부족했음
- Flutter install vs adb install -r 차이도 처음부터 알려줬어야 (세션 유실 의심으로 진단 logging까지 추가)
- `flutter install` 같은 도구 디테일은 매니저가 외부 정보로 미리 검증하는 게 사용자 시간 절약
